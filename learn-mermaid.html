What is Mermaid.js?

A powerful, text-based diagramming tool that creates professional diagrams using simple, Markdown-like syntax

Mermaid.js is an open-source JavaScript library that transforms plain text descriptions into polished diagrams and charts. Instead of dragging shapes around a canvas, you write a few lines of intuitive syntax, and Mermaid handles all the layout, positioning, and styling automatically. Think of it as Markdown for diagrams: just as Markdown lets you write formatted documents with simple text syntax, Mermaid lets you create complex visual diagrams without touching a mouse.

Mermaid supports a wide range of diagram types including flowcharts, sequence diagrams, class diagrams, state diagrams, entity relationship diagrams, Gantt charts, and more. It's natively supported in GitHub, GitLab, Notion, Confluence, and many other platforms, making it an ideal choice for developers who want their diagrams to live alongside their code and documentation.

Getting Started

Learn the basics in just 5 minutes

🔰 Your First Diagram

Every Mermaid diagram starts with declaring its type. Here's the simplest flowchart:

flowchart TD A[Start] --> B[Process] B --> C[End]

This creates a top-down flowchart with three nodes connected by arrows.

Subgraphs and Grouping

Organize complex diagrams into logical sections

As diagrams grow, they can become difficult to read. Subgraphs let you group related nodes into labeled containers, creating visual boundaries that mirror your system's actual architecture. They're essential for any diagram with more than 8-10 nodes.

Subgraphs can be nested to represent hierarchical structures like cloud regions containing availability zones, or organizational departments containing teams. You can also style subgraphs individually and create connections between subgraphs, not just between nodes.

flowchart TB subgraph Cloud["AWS Cloud"] subgraph VPC["VPC - 10.0.0.0/16"] subgraph Public["Public Subnet"] ALB[Load Balancer] NAT[NAT Gateway] end subgraph Private["Private Subnet"] App1[App Server 1] App2[App Server 2] DB[(RDS Database)] end end end User[Users] --> ALB ALB --> App1 ALB --> App2 App1 --> DB App2 --> DB

Key tips: give your subgraphs descriptive IDs and display labels using the subgraph ID["Display Label"] syntax. Keep nesting to three levels maximum for readability.

Sequence Diagram Deep Dive

Master the most popular diagram type for API and system documentation

Sequence diagrams are arguably the most useful diagram type for software developers. They show how components interact over time, making them perfect for documenting API flows, authentication handshakes, and microservice communication patterns.

Participants and Actors

Explicitly declare participants to control their order and give them display names. Use actor for human users and participant for systems:

sequenceDiagram actor User participant Client as Browser participant Auth as Auth Service participant API as API Gateway participant DB as Database User->>Client: Click Login Client->>Auth: POST /auth/login activate Auth Auth->>DB: Verify credentials DB-->>Auth: User record Auth-->>Client: JWT Token deactivate Auth Client->>API: GET /data (Bearer token) activate API API->>DB: Query data DB-->>API: Results API-->>Client: JSON response deactivate API Client->>User: Display data

Loops, Alternatives, and Notes

Use loop, alt/else, and Note blocks to show conditional logic, repetition, and annotations:

sequenceDiagram participant C as Client participant S as Server C->>S: Request with retry alt Success S-->>C: 200 OK else Rate Limited S-->>C: 429 Too Many Requests Note right of C: Wait and retry loop Retry up to 3 times C->>S: Retry request S-->>C: Response end end

Advanced Gantt Chart Features

Go beyond basic timelines with milestones, excludes, and sections

Gantt charts in Mermaid support much more than simple task bars. You can define milestones (diamond markers for key dates), exclude weekends or specific dates, mark tasks as done/active/critical, and organize work into sections. This makes them genuinely useful for sprint planning and release tracking.

gantt title Q1 Release Schedule dateFormat YYYY-MM-DD excludes weekends section Planning Requirements gathering :done, req, 2026-01-06, 5d Architecture review :done, arch, after req, 3d Sprint planning :done, plan, after arch, 2d section Development Core feature build :active, core, after plan, 10d API integration :api, after core, 7d Frontend polish :fe, after api, 5d section Release QA testing :qa, after fe, 5d Performance testing :perf, after fe, 5d Beta release :milestone, beta, after qa, 0d Bug fixes :fix, after beta, 5d Production release :milestone, prod, after fix, 0d

The excludes weekends directive automatically skips Saturdays and Sundays in the timeline. Use milestone for zero-duration markers that highlight key dates. Tasks can reference other tasks with after taskId to create dependency chains.

Common Mistakes and Debugging

Save time by avoiding these frequent pitfalls

Even experienced Mermaid users run into syntax issues. Here are the most common mistakes and how to fix them:

Special Characters in Labels

Parentheses, brackets, and quotes inside node labels will break your syntax. Always wrap labels with special characters in double quotes: A["Process (v2.0)"] instead of A[Process (v2.0)].

Missing Diagram Type Declaration

Every diagram must start with its type keyword (flowchart, sequenceDiagram, classDiagram, etc.). Forgetting this line is the most common cause of "Parse error" messages.

Indentation Matters in Some Contexts

While flowcharts are forgiving about whitespace, class diagram methods and Gantt chart tasks are sensitive to indentation. Use consistent spacing to avoid unexpected parsing failures.

Node ID Reuse

If you define A[First Label] and later write A[Second Label], Mermaid will use the first label. Node IDs are defined once; subsequent references don't need brackets. Just use A to reference an existing node.

Debugging Tips

When a diagram won't render, try these steps: First, check for a red error message in the preview, it usually points to the exact line with an issue. Second, comment out sections of your diagram to isolate the problem. Third, simplify complex labels by removing special characters temporarily. The SimpleMermaid editor shows errors in real-time, making it easy to catch issues as you type.

Integration Tips

Use Mermaid diagrams across your development workflow

One of Mermaid's biggest advantages is its wide platform support. Here's how to use your diagrams beyond SimpleMermaid:

GitHub and GitLab

Both platforms render Mermaid diagrams natively in Markdown files. Simply wrap your Mermaid code in a fenced code block with the mermaid language identifier. This means your architecture diagrams can live directly in your README, ADRs (Architecture Decision Records), or pull request descriptions.

```mermaid flowchart LR A[Feature Branch] --> B[Pull Request] B --> C{Review} C -->|Approved| D[Merge to Main] C -->|Changes Needed| A ```

Notion and Confluence

Notion supports Mermaid in code blocks natively. In Confluence, you can use the Mermaid plugin or embed diagrams as images exported from SimpleMermaid. For teams using Confluence, we recommend exporting as SVG for the best quality at any zoom level.

Documentation Sites

Popular documentation frameworks like Docusaurus, MkDocs (with the mermaid plugin), GitBook, and VuePress all support Mermaid diagrams. This lets you maintain a single source of truth for your diagrams in your documentation codebase, updated through the same pull request workflow as your code.

Presentations

For presentations, export your diagram as PNG from SimpleMermaid for maximum compatibility, or SVG if your presentation software supports it. The dark theme export works particularly well on dark slide backgrounds.

Best Practices

Create professional, maintainable diagrams

Writing good diagrams is a skill that improves with practice. These best practices will help you create diagrams that are clear, maintainable, and genuinely useful to your team.

  • ✅

    Use Descriptive Labels

    Instead of abstract identifiers like A, B, C, use meaningful names that describe what each node represents: User, AuthService, PostgresDB. A reader should understand the diagram without needing a legend. For connection labels, use action verbs: "authenticates," "queries," "returns" rather than generic terms like "connects to."

  • ✅

    Break Complex Diagrams

    If your diagram has more than 15-20 nodes, it's probably trying to show too much at once. Split it into focused views: a high-level system overview, a detailed authentication flow, a data pipeline diagram. Each diagram should answer one question clearly rather than trying to document everything in a single image.

  • ✅

    Use Consistent Direction

    Pick a flow direction that matches the mental model of your audience. Top-down (TD) works well for hierarchical processes and decision trees. Left-to-right (LR) is natural for sequential workflows and timelines. Whichever you choose, be consistent across your project's diagrams so readers can build intuition.

  • ✅

    Apply Color Coding

    Use colors to convey meaning, not just for decoration. Establish a consistent palette: green for success paths, red for error states, blue for external services, gray for optional steps. Document your color conventions so the whole team uses them consistently. Use classDef to define reusable styles.

  • ✅

    Group Related Items

    Use subgraphs to create visual boundaries that mirror your system's architecture. Group nodes by service, by team ownership, by deployment environment, or by security boundary. This makes large diagrams scannable and helps readers quickly find the section relevant to them.

  • ✅

    Label Important Connections

    Not every arrow needs a label, but critical transitions should be annotated. Decision branches should always show the condition (Yes/No, Success/Failure). API calls should show the method and endpoint. Data flows should indicate what data is being passed. When in doubt, add the label; a slightly verbose diagram is better than an ambiguous one.
