Mermaid for / Security

Threat models that live in your repo

Five ready-to-edit Mermaid diagrams covering RBAC, OAuth 2.0 PKCE, Kubernetes attack surfaces, supply-chain threats, and zero-trust architecture. Open any one in the editor, adapt it to your system, share via URL.

Jump to templates

Threat modeling fails in two predictable places: getting the diagram drawn at all, and keeping it current six months later. Most teams sketch a STRIDE diagram or attack tree during a kickoff workshop, paste a screenshot into Confluence, and watch it rot. By the time the architecture has shifted, no one wants to redraw it — so the documentation drifts away from the system it's supposed to describe.

Mermaid changes that arithmetic. The diagram lives next to the code as plain text. Trust boundaries, identity flows, attack vectors — all of them expressed in a few dozen lines of syntax that diff cleanly in pull requests. When the system changes, the diagram updates in the same commit. Reviewers see what changed in the model, not just that something did.

This page collects five diagrams that work as starting points for common security work. Click Open in editor on any example to start adapting it to your system — the diagram loads with full content via a shareable URL.

RBAC: User → Role → Permission

Class diagram

The smallest useful authorization model. Map users to roles and roles to permissions. Use as the structural backbone before drawing dynamic auth flows.

classDiagram
class User {
  +string username
  +Role[] roles
}
class Role {
  +string name
  +Permission[] permissions
}
class Permission {
  +string action
  +string resource
}
User --> Role
Role --> Permission

OAuth 2.0 PKCE Flow

Sequence diagram

A complete PKCE authorization-code flow with token exchange, refresh, and failure paths. Useful for review meetings when arguing about session handling and token storage.

sequenceDiagram
    participant U as User Agent
    participant SPA as SPA Client
    participant Auth as Authorization Server
    participant API as Resource API

    Note over SPA: PKCE Setup
    SPA->>SPA: Generate code_verifier
    SPA->>SPA: code_challenge = SHA256(verifier)

    SPA->>U: Redirect to /authorize
    U->>Auth: GET /authorize + code_challenge
    Auth->>U: Login form
    U->>Auth: Credentials
    Auth->>U: 302 Redirect ?code=xyz
    U->>SPA: Callback with auth code

    SPA->>Auth: POST /token + code_verifier
    Auth->>Auth: Verify SHA256(verifier) == challenge
    Auth->>SPA: access_token + refresh_token

    SPA->>API: GET /resource (Bearer token)
    API->>API: Verify JWT signature & claims
    API->>SPA: 200 OK

Why Mermaid for threat modeling

What you gain over Lucidchart, draw.io, or whiteboard photos.

Diff-able in pull requests

Reviewers see exactly what changed in the model — not just “here's a new screenshot.” Auditors trace decisions through git history.

No tool drift

Every developer renders the same diagram. No paid seats, no plugin install, no “the file format isn't compatible.”

Living documentation

The diagram lives in the repo with the code it describes. Change one, change the other — same commit.

Stays in your environment

Diagrams render client-side here — or in your repo's README. Sensitive architecture never leaves your network.

Tips for clearer threat-model diagrams

Patterns that have worked across the templates above.

  1. Mark threats as dashed red lines using classDef threat fill:#fca5a5,stroke-dasharray:5 so they're visually distinct from normal data flow.
  2. Group assets into subgraph blocks for trust boundaries (DMZ, internal, external). Boundaries are the highest-information part of any threat model.
  3. For attack chains, prefer sequence diagrams with Note over annotations — they read like an attack narrative, which is closer to how STRIDE conversations actually flow.
  4. Keep one diagram per concern: asset inventory, data flow, attack tree. Combining them into one mega-diagram makes the diff useless.
  5. Commit the .mmd file alongside your README. Link to a rendered version on SimpleMermaid for stakeholders who don't read markdown.
